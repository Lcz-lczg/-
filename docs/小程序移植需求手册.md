# 校园篮球比赛赛程管理系统 — 微信小程序移植需求手册

> 文档版本: 1.0  
> 日期: 2026-06-03  
> 原项目: 技术的校园篮球比赛赛程管理系统 (Spring Boot + Vue 3)

---

## 1. 项目背景

### 1.1 现况

原系统采用 B/S 架构（Spring Boot 后端 + Vue 3 前端），面向 PC 浏览器用户，已实现球队管理、球员管理、比赛报名、赛程自动编排、比赛结果录入、评论互动、新闻公告等功能。系统角色分为管理员、教练、球员、观众四类。

### 1.2 移植目标

将现有系统改造为**微信小程序**形态，保留核心业务功能，适配移动端交互，借助微信生态提升用户体验和传播效率。

### 1.3 移植策略

后端 API **尽量复用**，仅做兼容性调整（如微信登录、手机号授权等新增接口）。前端从 Vue 3 完全重写为微信小程序原生框架（或 Taro/uniapp 跨端框架，待技术选型确认）。

## 5. 技术方案建议

### 5.1 前端技术选型

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|:------:|
| **Taro 3.x + React/Vue** | 一套代码多端（H5/小程序/App）；生态成熟；可复用部分 Vue 逻辑 | 包体积稍大；调试有时不如原生 | ★★★★★ |
| **uni-app** | 多端能力强；Vue 语法接近原项目；插件市场丰富 | 性能略低于 Taro；编译问题较多 | ★★★★ |
| **微信原生 + TypeScript** | 体积最小；性能最佳；官方支持最好 | 无法复用原 Vue 代码；开发效率较低 | ★★★ |
| **WePY 2.x** | Vue 风格；轻量 | 社区活跃度下降 | ★★ |

**建议: Taro 3.x + React（或 Taro + Vue 3）**，兼顾开发效率和原生体验。

### 5.2 UI 组件库

建议 **TDesign WeApp**（腾讯官方，设计质量高）或 **Vant Weapp**（有赞，社区活跃）。

### 5.3 后端改造清单

| 改造项 | 说明 | 工作量 |
|--------|------|:------:|
| 微信登录接口 | 新增 `POST /api/wechat/login`，接收 code 换取 openid，返回 JWT token | 2d |
| 微信手机号解密 | 接收加密手机号数据，后端调用微信解密 API | 1d |
| 用户表调整 | `guanzhong`/`qiuyuanxinxi`/`jiaolian` 增加 `openid` 字段 | 0.5d |
| 内容安全 | 评论/昵称提交前调用 `security.msgSecCheck` | 1d |
| 订阅消息 | 对接微信订阅消息模板，后端保存 formId / 发送模板消息 | 2d |
| 图片上传优化 | 上传时压缩 + 自动缩略图生成 | 1d |
| API 分页统一 | 确保所有列表接口支持 `page`/`pageSize` 分页参数 | 1d |
| 小程序码生成 | 比赛分享/球队分享生成小程序码 | 1d |
| **合计** | | **~9.5d** |

## 9. 风险与注意事项

1. **微信审核风险**: 小程序涉及「体育赛事」类目，需确认是否需要相关资质（《增值电信业务经营许可证》等）。建议提前查阅微信小程序类目要求。
2. **内容安全红线**: UGC 内容（评论、昵称）必须接入微信内容安全 API，否则无法通过审核。
3. **登录方式限制**: 微信小程序不允许强制用户授权手机号，需提供其他登录方式兜底。
4. **富文本编辑器**: 小程序端富文本编辑能力弱（moveable 等组件不支持），建议管理员复杂编辑仍通过 PC 端完成，小程序端只做简单文本录入。
5. **赛程编排算法**: 原贝格尔编排算法的 Java 实现可以直接复用，无需改动。
6. **API 兼容性**: 原后端接口返回格式统一为 `{code: 0, data: ..., msg: ...}`，小程序端直接对接，无需改动已有接口签名。
